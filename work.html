SOPHIA — Knowledge Retrieval Agent
A 70+ member team navigating 100+ client documents leaned on team leads for every "where is this / what's the process" question — slowing everyone down and creating heavy dependency.
+

◆ The problem

The account carried heavy client documentation — over 100 files spanning SOPs, policies, and buying guides. Finding the right answer meant either hunting through SharePoint or, more often, messaging a team lead. That made leads a bottleneck and slowed the whole team's response time.

◆ What I built

An AI knowledge-retrieval agent integrated with the entire documentation base — SOPs, policies, buying guides, and internal documents. Anyone on the team can ask a question in natural language and get an answer in seconds, with the agent also pointing them to the source file and helping locate documents on the team SharePoint.

Cut lead dependency · seconds to answer vs. waiting on a person

Supplier Communication Automation
Every sourcing project means chasing 5–10 suppliers over email — intent to participate, multiple RFP-round reminders, follow-ups. Across active projects, this manual outreach became a constant drain on the team.
+

◆ The problem

In sourcing, every project requires repeated email contact with each supplier — confirming intent to participate, sending reminders before each RFP deadline (often multiple rounds), and following up. With a minimum of 5–10 suppliers per project across many live projects, this communication load became a genuine hassle for the delivery team.

◆ What I built — a multi-layered, end-to-end flow

  • 1Ingest. After the kick-off call, the requirement template is uploaded to a SharePoint folder. That upload is the trigger: the flow scans the file, extracts all project and supplier details, and writes them to a SharePoint list — a new row per supplier, so tracking is clean from the start.
  • 2Check & act, daily. Every day the flow reviews each supplier row against four checkpoints — project start date, current stage with that supplier, follow-up count, and next follow-up date — then takes the right action. (e.g. project just started, intent not yet confirmed, follow-up count 0, due today → it emails the supplier, copies the project owner, and includes a short project brief.) It then auto-updates all four fields, so the loop is fully closed.
  • 3Monitor replies. The flow watches the inbox and routes responses: confirmed intent → updates status in the list; deadline-type questions → answers automatically from the kick-off template; technical questions it can't answer → posts a notification to the project owner's Teams to pick up.

◆ Impact

1 FTE · 1,920 hours saved annually

This automation, standalone, accounts for a full FTE of capacity returned to the team.

Spend Intelligence · Agentic AI
Opportunity Assessment Agent
Spotting savings and governance opportunities buried in a quarter's PO spend — clubbing, consolidation, off-process buying — was slow, manual analysis that was easy to miss at scale.
+

◆ The problem

A lot of value hides inside raw PO spend: spend that bypasses procurement, the same product bought across multiple projects, repeat low-value orders kept under threshold. Finding it by hand across a full quarter of spend is slow and inconsistent.

◆ What I built

An agent that analyzes a quarter's PO spend file and surfaces three opportunity types:

  • 1Below-bidding routing. Finds below-bidding spend that could be clubbed with an above-bidding project — bringing more spend under procurement governance and lifting both savings and managed spend.
  • 2Above-bidding consolidation. Detects the same product/service raised across multiple projects and flags it for consolidation — reducing effort, creating one high-value project as a negotiation lever, and consolidating suppliers.
  • 3Below-bidding consolidation. Identifies the same product/service ordered repeatedly under threshold (to bypass procurement) so it can be clubbed and brought back under governance.

◆ Inputs

Two files: the quarterly PO data (spend, supplier awarded, product/service, category) and a category–country mapping that fills any gaps in the PO data.

◆ Impact

More spend under governance · stronger negotiation leverage · less manual analysis

Sourcing Intelligence · Agentic AI
Supplier Identification Agent
Finding the right suppliers for a new project meant manually cross-referencing repositories, category mappings, and past client work — repetitive lookup work for every sourcing event.
+

◆ The problem

Identifying candidate suppliers for a project required pulling together several data sources by hand — internal and client supplier repositories, category mappings, and prior client project history — for every new sourcing event.

◆ What I built

An agent integrated with four data points — the client supplier repository, our supplier repository, client project data, and category mapping. You hand it the SOW; it maps the scope to category requirements, finds suppliers operating in that category and country, and structures the output two ways:

  • 1Invite-ready suppliers from the repository that fit the project.
  • 2Proven suppliers that have already worked on similar scope with the client.

Both draw from the same sources — the split just makes the answer immediately usable.

◆ Impact

Supplier shortlists in seconds · grounded in real repository & history data

Spend Intelligence · Agentic AI
Catalogue Opportunity Identification
Recurring, catalogue-suitable purchases were still being bought manually — but spotting which products/services should become catalogue items required digging through spend data category by category.
+

◆ The problem

Many recurring purchases are good candidates to move into a catalogue (faster, automated buying) — but identifying them means knowing which categories are catalogue-conducive and then analyzing buying channels and demand across the spend.

◆ What I built

An agent that analyzes PO spend and client project reports against an integrated reference of catalogue-conducive categories, then filters the spend to those categories and assesses two things:

  • 1Buying channel — is it already catalogued, or bought manually?
  • 2Demand — does the product/service show recurring demand?

The output is a shortlist of demand that can be catalogued and automated, which the procurement team then actions.

◆ Impact

Surfaces automation-ready buying · less manual catalogue analysis
